# 알려진 한계

MVP 범위에서 의도적으로 수용한 제약과 그 이유, 해소 경로를 정리합니다.

## 1. 데이터 레이어 — JSON 파일 저장소

**현상**: Prisma 스키마(`prisma/schema.prisma`)는 정식 작성되어 있으나, 런타임은 `data/db.json` 기반 JSON 파일 저장소로 동작합니다.

**이유**: 개발 샌드박스에서 Prisma 쿼리 엔진 바이너리 다운로드(binaries.prisma.sh)가 네트워크 정책상 차단되어 있었습니다.

**영향**:
- 동시 쓰기 보장 없음 — 단일 사용자/단일 노드 전용
- 트랜잭션 없음 — 문서 반영 중 서버가 죽으면 부분 반영 가능
- 데이터가 커지면 전체 파일 직렬화 비용 증가

**해소**: README의 "데이터 레이어에 관한 안내" 절차대로 Prisma로 전환. `store.ts` 함수 시그니처와 `src/types/models.ts` 타입이 스키마와 1:1이므로 호출부 수정 불필요.

## 2. Mock AI Provider의 추출 한계

**현상**: API 키 없이 동작하는 Mock Provider는 정규식 휴리스틱(HH:mm 시간 패턴)으로만 프로그램을 추출합니다.

**영향**:
- 담당자, 장비, 의존성, 의사결정권자 추출 불가
- 표 형태·서술형 문서에서 정확도 낮음
- 신뢰도는 항상 낮게(≤0.6) 표시되어 사용자 검토를 강제

**해소**: `.env`에 `ANTHROPIC_API_KEY` 설정 시 자동으로 Claude Provider로 전환.

## 3. 직접 입력 대응안의 고정 평가

**현상**: 선택지 대신 자유 입력한 대응은 Mock 모드에서 고정된 중립 효과(소폭 감점 방지 수준)로 평가됩니다.

**이유**: 자유 텍스트의 효과 산정은 LLM 판단이 필요한 영역입니다.

**해소**: Phase 2에서 Claude 기반 평가 연결 (`decision-evaluation.schema.json` 스키마는 이미 정의됨).

## 4. 시간 진행 방식

**현상**: 시뮬레이션 내 시계는 상황 트리거 시각으로 점프하는 방식이며, 연속적인 실시간 진행이 아닙니다.

**영향**: "상황과 상황 사이"의 자율 판단(예: 여유 시간에 선제 점검)은 표현되지 않습니다.

**해소**: Phase 3 실시간 모드.

## 5. 연쇄 깊이 1 제한

**현상**: 연쇄 상황의 선택지는 새로운 연쇄를 유발하지 않습니다.

**이유**: 초기 구현에서 연쇄가 무한 증식하는 문제가 확인되어, MVP에서는 깊이 1로 고정했습니다.

**해소**: Phase 3에서 설정 가능한 깊이 + 연쇄 트리 시각화로 확장.

## 6. PDF 리포트 미구현

**현상**: 결과 리포트는 화면 + Markdown 내보내기만 지원합니다.

**해소**: Phase 1에서 서버사이드 PDF 렌더링 추가. Markdown 구조가 이미 리포트 전체를 담고 있어 변환 비용은 낮습니다.

## 7. 인증/멀티유저 없음

**현상**: 로그인 없이 모든 프로젝트에 접근 가능합니다. 데모/개인 사용 전제입니다.

**해소**: Phase 1 워크스페이스 도입 시 함께 해결.

## 8. 백그라운드 문서 처리의 내구성

**현상**: 문서 처리 파이프라인은 fire-and-forget Promise로 실행됩니다. 처리 중 서버 재시작 시 해당 문서는 `extracting`/`analyzing` 상태에 머뭅니다.

**완화**: 문서 목록에서 오류 상태와 메시지를 노출하며, 같은 파일을 재업로드하면 새로 처리됩니다.

**해소**: Phase 1 큐 기반 처리 + 재시도.

## 9. 점수 모델의 단순성

**현상**: 점수 효과는 선택지에 정적으로 정의된 값 × 난도 배율입니다. 상황 간 상호작용(예: 피로도가 높으면 실수 확률 증가)은 리포트 집계에서만 반영됩니다.

**해소**: Phase 3 적응형 난도와 함께 동적 효과 모델 도입 검토.

## 10. 브라우저 호환성

**현상**: 최신 Chrome/Edge/Safari 기준으로 개발했습니다. 구형 브라우저(IE 등)는 지원하지 않습니다.
